RAMI 4.0 (Reference Architectural Model Industrie)

Glossary

Asset
Some thing (maybe of value) that you have (maybe own) and would like to use. Could be hardware, like a machine or an assembly line. But also a cake, a car, a cat. And also virtual things like software, a database, an archive, an idea. Secret agents also have assets (dt. Informanten).
Administration Shell, Asset Administration Shell (AAS) (dt.: Verwaltungsschale)
A well defined, thin, general interface that is put around (very different) assets to allow for a unified communication. It does not add functionality or logic to the asset itself. Can be used to make an asset a self-describing component, that can provide e.g. its name, its properties, its state and its capabilities (services) in a so called Manifest. The Manifest acts as a clearly locatable table of contents for all information, data and functions in the Administration Shell.

Concept

One part of the RAMI is that so called Assets communicate with each other via lightweight Asset Administration Shells (AAS).
An asset is the owner of exactly one AAS, but can consume an arbitrary number of other AAS. Each AAS comprises a number of submodels (SM), basically different aspects of one and the same asset. Each submodel defines Properties, Events, and Operations.

Implementation

One of many many ways to implement an AAS is to define them using JSON files. Each submodel is defined in a separate JSON file. One example for a definition of a X-ray machine can be found here.

The JSON file for a submodel has to comply with a given schema. It is a definition of fields (mandatory, optional) and types (e.g. `int`, `float`, `bool`) that are allowed in the submodel file. An example schema can be found here.
